受到 fitlog 的启发，thexp 实现了一套和 git 相配合的试验管理方法，该模块位于 thexp.frame.experiment。

本节的示例位于 examples/7.exp.py，但在使用该模块之前，你可能需要先接受 thexp 对路径组织的相关设定。这也是我个人认为的试验中间数据存储的最佳实践。

TIP

当然，如果您有更好的方式，欢迎 Issue ，我会选择使用更合适的方式来组织试验目录。

# 路径组织

试验、程序项目、课题...这些概念之间的关系应该是什么样的呢？做试验的时候可能都不会注意这些概念，但当有了对试验组织的需求后，就需要明确这些关系。在我复现过的一些实验中，很多都是将中间数据（模型、日志等）存储放在项目一个文件夹下，其命名由各个超参数值构成，我在最初也尝试过这种构建方法，但最终选择抛弃不用。

TIP

太长的参数不利于在服务器上切换目录，固定的参数对应的多次重复试验也无法很好的进行分辨和记录。

在thexp的理解中，一般的试验流程为，根据某一个课题（project），创建一个项目（repository），在项目中一步步的设计多个实验（experiment），而每一个实验都可能有多次运行代码的需求，一次代码的运行过程中可能会尝试多个实验或重复验证某实验，也即多次试验（test）。此时每一次运行代码就需要在 git 中进行一次提交，而一次代码运行期间的多次试验则通过序号来进行标识，确保全局 试验ID 不存在重复。

为什么以代码运行为单位而不是试验为单位进行提交？

Python 在代码运行期间，使用的代码为运行时候获取到的代码。因此如果以试验为单位进行提交，中途可能会因为项目代码变更导致试验和代码无法对映。（仍然存在代码运行中间才新导入某个文件的情形，这种情况没有很好的办法进行判断，但在具体操作中很少能出现这种事故，对其有一定概念即可）。

最终，thexp 确定的目录结构如下：

{可自由设置的存储根目录}/{项目名}.{项目hash}/{   实验名   }/{本次提交的试验次数}.{提交hash}/
--------------------   -------------------   -----------  ---------------------------
                          项目目录            实验级目录           试验级目录
1
2
3

TIP

没有课题级别的目录，课题在这里的表现是可能具有相同的项目名，但是本地存储路径不同，也即，在 git 中表现为不同的 repository。而 thexp 会对不同的项目分配一个两位ID对相同的项目名进行区分。（每一位均为随机的数字或字母，因此重复概率为 1/1296）

# 基本使用

WARNING

首先确认你安装了 git，不论你是否使用 thexp 进行试验管理，我都强烈建议使用 git 进行项目管理

from thexp import Experiment

exp = Experiment("expname") # 程序运行过程中第一次创建 Experiment 对象会产生一次提交，之后重复创建不会产生提交，但是获取到的试验目录会不同
print(exp.make_exp_dir("explevel")) # 分配 experiment 级别的目录
print(exp.makedir("testlevel")) # 分配 test 级别的目录


# exp.end() # 可选项，否则Experiment会在程序退出前自动调用该方法结束试验

# 使用插件记录一切

凡走过，必留下痕迹。 ——艾德蒙·罗卡

在日志输出、模型存储、数据记录等过程中存储的数据在试验量级变大后并不是很好管理，因为各个试验对这些功能的存储位置、甚至是否使用等情况可能各不相同。换个角度思考，我们缺少对这些存储数据的元数据记录。

thexp 因此提出了插件这一概念。在试验中产生的任何存储过程或其他各种各样与试验核心内容无关的部份，都被看做是插件（plugin）。thexp 推荐在试验过程中将所有使用到的插件通过 Experiment 进行记录。这会有助于我们对多个试验进行分析。

比如，你可以存储超参数

from thexp import Params

params = Params()
exp.regist_plugin("params",dict(
    _param_hash = params.hash(), 
    data = params.inner_dict.jsonify()
))

from thexp import Logger

统一带来简约，简约提升效率

实际上，利用Experiment 的目录组织，即使不通过 regist_plugin 方法存储元数据，你也能够通过各种判断来完成这些信息的获取。但将所有的数据统一到 info.json 中，并通过 thexp 中提供的 分析功能，你可以用更统一的方式处理不同的各类插件。

# 试验分析与查找

thexp.analyse 中提供了对试验的查找和分析功能，建立了Summary/Project/Exp/Test 四个粒度的 Viewer并将其联结起来。

你可以很方便的遍历本地所有的试验：

# 代码归档和重置

在之前对试验目录组织的说明中，如果你熟悉 git ，可能已经想到了利用提交来对代码进行恢复。thexp 确实实现了这一功能。

你可以对某次的代码进行打包（基于 git archive）

from thexp.analyser.expviewer import SummaryViewer
from thexp.utils.gitutils import archive

viewer = SummaryViewer()
res = viewer.find('0002.41505917')
exp = archive(res) # type:Experiment
1
2
3
4
5
6

也可以将代码恢复到某一次试验（基于 git checkout/git branch/git reset)

from thexp.analyser.expviewer import SummaryViewer
from thexp.utils.gitutils import reset

viewer = SummaryViewer()
res = viewer.find('0001.8e7f52e2')[0]
exp = reset(res) # type:Experiment
1

3
4
5

可恢复性

打包和重置操作在 thexp 中同样是被当做是一次试验操作，对打包操作来说，这一行为为打包提供了存储目录；而对重置操作，这意味这你可以通过重置时的试验名，随时恢复到重置前的状态。

打包和重置操作的实验名分别为 Archive 和 Reset。

# 多级配置

thexp 同时还提供了一个配合 git 的全局配置变量， glob，提供了user、repository、exp三个级别的配置，其中：

  • user级别的配置以 json 格式存储于$home/.thexp/config.json，会在任何 thexp 相关包导入的时候进行初始化；
  • repository级别的配置通过 git 存储在当前项目下的 config 中；默认会设置当前目录下的.thexp/为试验存储目录，此外，会对 .gitignore 文件进行检查，将 .thexp/ 文件夹添加进去。
  • exp是运行时配置，相当于全局的 Params，任何在运行时的配置设置均为运行时配置。

使用方法：

from thexp import glob

glob['a'] = 4

# glob.add_value("key",'value','user')
# glob.add_value("key",'value','exp')
# glob.add_value("key",'value','repository')

from pprint import pprint
pprint(glob.items())
